I/O

java中的file对象实际上只是一个抽象的概念，FileDescriptor才是真正意义上指代file的对象。

输出流负责写入数据，输入流负责负责读取数据。
相比于输入输出流，缓冲流有一块区域叫做缓冲区，缓冲区用于数据流的缓存，如果向缓冲流中读取数据，首先是从缓冲区中读取数据，如果此时的缓冲区为空，系统会自动的从磁盘中读取数据，尽可
能的填充满缓冲区，此时从设备中读取的数据并不会直接发送给外部设备而是暂存于缓冲区中。
在处理输入流的时候一般都是一个一个字节的读取输入流内的数据，不过也可以通过一段缓冲区来指定读取的字节数:

byte[] buffer = new buffer[1024];
while(inputStream.read(buffer) != -1){
    outputStream.write(buffer);
}

可以把文件想象出一个用于装字节的容器，FileInputStream 想像成一个水管用于抽取文件中的字节s，对应的FileOutputStream就是一个专门用于向容器中注入字节s
哎，其实主要是根据命名去理解stream太蛋疼了,文件输出流不应该是用于抽取文件字节 文件输入流不应该是用于注入字节的吗?咋刚好相反呢 ?

okhttp 源码

整个okHttp源码最为核心的部分就是Interceptor了,它负责了实际的网络请求、错误重发、缓存、压缩、解析等一系列的实质性的工作，而其中的每一项工作都是具有独立的Interceptor来完成对应的工作.也就是一个典型的责任链模式,而每一个Interceptor所在chain的位置决定了它的功能,例如请求服务器数据 和 解析服务器返回的数据就应该在此责任链的尾部.

在阅读源码的过程中,着重了解okHttp的 request缓存策略, Interceptor实现机制以及Interceptor的Chain机制, dispacher下发Call的机制, 路由原理, 连接池如何减少延迟,HttpEngine是如何处理核心事务(request->response)以及学一学如何用一门语言来实现传输协议(不限于java来实现HTTP/HTTPS,例如实现MQTT协议呢?)

####自言自语
其中的StreamAllocation负责协调了Connection/Stream/Call三个对象之间的逻辑(因为涉及到connection, 所以这个类引入了Route). 因为在发送request最终会走到httpEngine的sendRequest方法,在这个方法里面会根据不同的状态执行不同的生成response的操作,而正常情况下会走获取服务器的response分支,而此分支的重点就是在connect()函数,它返回的是一个HttpStream对象.

connect的函数实现是转交给了StreamAllocation的newStream()函数, 在newStream中最先做的是初始化一个RealConnection对象出来, 而剩余的工作就是判断http的版本,是初始化2.0版本的HttpStream对象还是以下版本的HttpStream对象。 有一个疑问是Stream对象到底是什么的抽象。我一直理解的是像FileInputStream对象一样是一个数据流的抽象,也就是说只应该关心数据(2进制)的读取or写入的逻辑才对, 而这个httpStream同时囊括了写入和读取的操作以及StreamAllocation对象的实例是通过new StreamAllocation(client.connectionPool(), createAddress(client, request)); 来初始化的,这个request是最初状态的request.

总结一下同步请求request的流程: (难点在于理解Sink 以及Stream的一个抽象, stream实际上是一个利用了OkIO对socket处理封装而成)
client.newCall(syncRequest).execute(), 首先是获取到一个RealCall的对象,是对发送请求行为的一个抽象, 接着从execute()作为入口, 创建出一个InterceptorChain对象,负责整个的请求流程中的实质性工作, 然后就是启动这个责任链了,当然会先去判断此责任链上的interceptor的个数以及各自所负责的任务是啥,咱先默认没有interceptor,直接调用getResponse()来获取response, 首先要检查我们发送出去的request是否携带了body体,如果不为空的话我们需要重新复制一份request出来(包括重新设置header的一些属性), 然后就是初始化一份httpEngine对象出来,此引擎对象是真正处理http事务的master. 紧接着就是进入一个while(true)循环, 第一步就是通过engine向服务器发送request, engine内部的发送request的逻辑为: 0. 将用户发出的request格式化成机器读的request 同时还设置了一些默认的header属性以及cookies, 1. 从client获取一下负责缓存的对象, 看看这个request有没有之前缓存下来的response 如果有的话就好说了,将其作为一个候选返回对象(并非一定可用,如果在可用时间内则可以作为返回对象否则还是需要刷新,涉及一个有效期检查的操作), 接着还需要判断网络环境、是否开启缓存设置来进一步筛选候选返回对象, 2. 常规的连接服务器来获取response.

而负责跟服务器建立连接的逻辑放到了connect()函数内部，返回的是一个httpStream对象。在connect内部中实际上就是通过streamAllocation初始化出一个httpStream对象. 紧接着就是把这个stream对象和HttpEngine绑定在一块, 然后检查一下body体的大小, 超过2g不给发送。 最终利用httpStream的writeRequestHeaders方法把request的header发送给服务器. (个人疑惑的是Sink这个类是什么的一个抽象, 类似一个socket?)
在完成了上述的send操作, 紧接着就是读取response操作了, 如果服务器尚未准备好内容返回给请求方则会进入到catch 内部, 进而break掉当前的操作重新进入循环体, 直到服务器返回了response, 可见这个是一个阻塞操作, 不应该丢在主线程内去执行。
从服务器读取response操作有一个奇怪的地方, 会判断缓冲区的requestBody是否为空不为空的话发送给server? 为啥在getResponse的时候才发送body体内容呢?

如何记录一个request的请求cost掉的时间? 在send的时候httpStream会纪录请求时间, readResponse会记录finishRequest的时间,这个时间差就是判断是否连接超时的一个依据。

在读取并返回response的时候, 还是对httpStream的操作, 通过OkIO库对字节流的格式化生成一个network读的response, 后续的工作肯定就是对机器读的response格式化成用户友好型的response了,例如格式化response的header 的cookies, 如果有缓存的话直接用缓存的response没有缓存才用network的response来响应请求。

以上是最简单情况的同步request, 那如果是有interceptor的话, 又会是什么样的流程呢?
Interceptor其实就是对网络请求的一个封装而已。上述的都提到过了, 而interceptor 只不过是提供给开发者一个更为灵活的可自定义的处理方式来取代上述默认的处理request->response的逻辑。

####异步请求
dispacher: 是异步的核心类
持有一个ExecutorService, 三个队列(正在运行中异步call队列,正在运行中同步call队列,准备运行异步call队列)。把callback的引用封装成一个AsyncCall对象压到准备运行异步call队列中去,由executor来调度线程来启动call的execute()函数, 在此函数内部, 依旧是通过上述的getResponseWithInterceptorChain()方式获取到response结果(子线程中同步获得), 如果在执行期间调用了cancel那么将会回调callback引用的onFailer()函数, 否则的话就是回调onResponse()函数.流程也比较简单。 主要是理解executor是如何调度线程从队列中获取Call来执行这个过程要明白.

关于DCL的笔记

装逼用的是说DCL ,其实就是double check lock,如下代码:

以上就是为了解决并发环境下获取单例实例的一个写法, 第一个判空是为了避免不必要的同步代码操作，第二个判空则是为了保证只执行一次实例操作。
当执行到了 mInstance = new Singleton(); 也就是实例化一个类对象的时候，粗略的看来是完成了三个操作:(可参考 虚拟机遇到new时内存分配)

* 为类实例分配内存;
* 调用类默认构造函数;
* 将分配好了的内存空间地址指针指向类实例(此时的mInstance == null就不成立了)

(由此可得,类对象是否成功的实例化的依据是类引用指向了内存指定的地址而不是调用了构造函数)
由于在JDK1.5之前的JMM(java memory model)中规定,第二条和第三条的执行顺序不可定,如果线程1是1,2,3的顺序执行实例化操作,在任何一个步骤中有外来线程调用getInstance()都不会有问题的;但是如果线程1是1,3,2的顺序来执行实例化操作的话,外来线程调用到了getInstance()函数,此时的构造函数并没有调用到而只是分配到了内存空间,此时的外来线程如果获取到instance从而去执行某个类函数的话就会因尚未执行构造函数导致异常的发生.
解决方法就是给mInstance添加volatile修饰,这样可以保证读取的instance都是从主内存中获取的，保证了读的一致性。
