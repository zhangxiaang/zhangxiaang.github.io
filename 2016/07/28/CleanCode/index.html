  • 对于接口的命名,添加”I”前缀也许是比较中规中矩的做法，但是建议是能够脱离这个前缀。实现这些接口的类比较好的方式添加一个”Impl”后缀。
  • 取名的时候尽量不要 搞笑/装深沉，即使要装，也要装的好。就像之前的蜻蜓FM的欺骗友盟统计的Zeus和Oedipus函数，给人一种很hack但是不知它们是干嘛的感觉(人家就是不想让你知道…)

  • 多尝试用用计算机专业术语来作为取名的一个源泉吧，毕竟只有程序员才会读你的code。

  • 函数章节:

    • 每个函数只讲一件事儿,并且做好它
    • 讲完一件事儿要么结束要么带你进入到下一件事儿。
    • 不要这件事儿没讲完就跑去讲另外一件事儿。
    • 在if/else/while语句中，强迫自己使用一行代码也就是一个函数来表述要做的事情。例如: if(condition) buyFood();
    • 函数体内的抽象层级应该保持一致，否则就应该分成多个函数;例如，一个函数里先判断了obj的类型然后根据类型去执行不同的计算结果，这很明显就是两个层级，应该分成 判断类型+计算结果 两个层级函数。
    • 函数取名的时候别惧怕长名称，一个具有描述性的长名称会比一个短小却让人莫名其妙的名称要好得多。
    • “事件函数” -> 参数是为了修改系统状态，而无输出参数，需要慎重的取函数名。
    • 对于目的是”转换内容”的函数，输入参数应该是需要被转换的对象，返回值是转换后的对象。例如 File transform(File in);
    • 对于传入参数带有boolean变量的函数其实本质上是不好的，因为在函数体内会因为boolean的不同而执行两个操作，违反了”只做一件事儿”的原则。尝试把这个函数体一分为二吧，避免在参数列表出现boolean变量。
    • 在参数列表的长度长达三个或以上的时候，就应该考虑某些参数是否可以封装为类了，例如drawLine(int origin_x,int orgin_y,int target_x,int target_y,int line_width);看起来很难受，可以封装为drawLine(Ponit origin,Point target,int line_width); 会舒服很多;android中这个问题很严峻，在scroll函数中干嘛搞出那么多的参数呢?
    • 函数名最好的形式就是 动词+关键字 动词体现出你要做什么，关键字体现出你要操作谁?
    • 在函数体内不要编写try/catch，尝试吧Exception throw出去。
    • 写代码和写文章一样，前期把你想的都一股脑抛出来，然后再修改删除语句，标点符号，直到文章/代码变成你心目中的样子(好读 好用)，就是大功告成啦。

  • Demeter定律: 模块不应该了解到它所操作的对象的内部结构

    • 对象应该隐藏数据，暴露的是结构。
    • 一个类的方法只应该可以调用:该类对象的方法;在f函数内创建出来的对象的方法;作为参数传递到该方法时也可以调用这个参数的方法;最后一个就是该类中的成员变量的对象的函数。仅此而已。

  • 类的内聚

    • 类应该只有少量的实体变量，类中的每一个方法都应该是操作到这些实体变量的。从而达到类的高度内聚。
    • 很多时候我们喜欢把一个函数的参数的变量提取出来，作为一个类的变量。这个做法究竟值不值得呢?
      • 如果说这个函数的参数在很多函数中都会被用到，例如说Context对象，提升为类的实体对象是很值得的，大大提升了类的内聚度；不过如果说只是一个很普通的参数，例如一个int length参数，只有在类中的计算面积的函数中用到，那么是完全没有必要提升为实体对象的，如果所有函数都把诸如length类型的参数提升为实体对象，那么造成的后果就是类显得无比的臃肿，每个实体对象都只是在极个别的函数中被引用到。
